# Questionnaire Builder — Agent Instructions

React 19 + TypeScript SPA for building, sharing, and analysing questionnaires using **SurveyJS**. See [PLAN.md](Docs/Completed/PLAN.md) for full project history and [Docs/SurveyJS/](Docs/SurveyJS/README.md) for SurveyJS component docs.

---

## Commands

All commands can be run from the repo root (workspace delegates to `Frontend/`) or from `Frontend/` directly.

```bash
pnpm dev            # Vite dev server (http://localhost:5173)
pnpm build          # tsc -b && vite build
pnpm lint           # ESLint on Frontend/src/
pnpm lint:fix       # ESLint --fix
pnpm format         # Prettier on Frontend/src/
pnpm test           # Vitest (unit + integration)
pnpm coverage       # Vitest with coverage
pnpm test:e2e       # Playwright
```

Use `pnpm`, not `npm` or `yarn`.

> **Repo layout**: Frontend code lives in `Frontend/`. The root `package.json` and `pnpm-workspace.yaml` delegate all scripts to the `Frontend` package. To add dependencies, run `pnpm add <pkg>` from inside `Frontend/`.

## Feature Overview

The app has three user flows:

```
1. Builder  →  /questionnaires/:id/json
              Raw JSON editor → saves surveyJson to backend via PATCH

2. Respondent  →  /take/:id          (shareable public URL)
              Survey renderer reads surveyJson → onComplete → POST /responses

3. Owner  →  /questionnaires/:id/results
              Fetches responses[] → VisualizationPanel renders charts
              Optional: export to PDF via survey-pdf
```

**Shareable URL pattern**: `window.location.origin + /take/${id}`  
Copy-to-clipboard button lives on the questionnaire list page (`/questionnaires`).

```

  __root.tsx                         # layout (nav bar)
  index.tsx                          # redirect → /questionnaires
  questionnaires/
    index.tsx                        # list: title, single Edit button (→ /json), shareable link, results link

    share.tsx                        # /questionnaires/share — copy shareable URLs for all questionnaires
    $id/
      json.tsx                       # /questionnaires/:id/json — split-view: JSON editor (left) + live SurveyRenderer preview (right)
      results.tsx                    # SurveyJS Dashboard + PDF export + Raw Responses table
  take/
    $id.tsx                          # Public respondent view (no auth required)
```

Never edit `Frontend/src/routeTree.gen.ts` — it is auto-generated by the Vite plugin.

## SurveyJS Integration Conventions

> See [`.github/instructions/surveyjs.instructions.md`](.github/instructions/surveyjs.instructions.md) for detailed patterns.

- **`QuestionnaireType.surveyJson`** (type `object`) stores the raw SurveyJS JSON schema. The old `questions: Question[]` array is superseded by this field.
- **JSON editor** saves via `PATCH /questionnaires/:id` with `{ surveyJson: <parsed JSON> }`.
- **Survey Renderer** submits via `survey.onComplete.add` → `POST /questionnaires/:id/responses`.
- **SurveyJS components are not SSR-safe** — this is a Vite SPA so no issue, but do not attempt server rendering.
- Always import SurveyJS CSS explicitly (see instructions file).
- `survey-creator-react` and `survey-creator-core` have been **removed** — do not re-add them.

## Key Conventions

### Routing
- Routes live in `Frontend/src/routes/` — the Vite plugin auto-generates `Frontend/src/routeTree.gen.ts`. **Never edit that file.**
- Root layout: `Frontend/src/routes/__root.tsx`. New routes follow the same export pattern as `Frontend/src/routes/index.tsx`.

### State / Queries
- Follow the query-key factory pattern in [`Frontend/src/hooks/useQuestionnaires.ts`](Frontend/src/hooks/useQuestionnaires.ts).
- Mutations must call `queryClient.invalidateQueries()` on success to refresh lists.

### Forms
- Simple forms use controlled React state or `@tanstack/react-form`.
- Validators receive `{ value }` (not the raw value) — TanStack Form v1 API.

### API Layer
- Add typed functions to the relevant file under `Frontend/src/api/`. Follow the Omit pattern for payloads (`Omit<T, 'id' | 'createdAt' | 'updatedAt'>`).
- Auth token is read from `localStorage` by the Axios interceptor — no manual headers needed.
- Set `VITE_API_BASE_URL` in `Frontend/.env` to point at the backend; defaults to `/api`.
- Response API lives in `Frontend/src/api/responses.ts` — `submitResponse(id, data)` and `getResponses(id)`.
- Response hooks live in `Frontend/src/hooks/useResponses.ts` — follow the same query-key factory pattern as [`useQuestionnaires.ts`](Frontend/src/hooks/useQuestionnaires.ts).
- The `/take/:id` route calls `getQuestionnaireType` without auth (public); responses are submitted without auth.

### UI Components
- Reusable primitives live in `Frontend/src/components/ui/`. Use the `cn()` helper from `Frontend/src/lib/utils.ts` for className merging.
- Use CVA variants + `asChild` (Radix Slot) pattern — see [`Button.tsx`](Frontend/src/components/ui/Button.tsx).
- Tailwind v4: theme values are CSS variables (e.g. `var(--color-primary)`) defined in `Frontend/src/index.css`.

### TypeScript
- Strict mode + `noUnusedLocals` + `noUnusedParameters` — unused variables cause **build failures**.
- `QuestionnaireType.surveyJson` is `object` (raw SurveyJS JSON). The old `questions: Question[]` array is **not** used for SurveyJS-powered forms.
- Import alias `@/` maps to `Frontend/src/`.

## Testing

- **Unit/integration**: Vitest + Testing Library. Files in `Frontend/src/test/`. Globals (`describe`, `it`, `expect`, `vi`) are available without importing.
- **Setup**: [`Frontend/src/test/setup.ts`](Frontend/src/test/setup.ts) imports `@testing-library/jest-dom`.
- Each test file should wrap components with `QueryClientProvider` using a fresh `QueryClient({ defaultOptions: { queries: { retry: false } } })` to avoid cache pollution and timeout flakiness.
- Mock API calls with `vi.mock('../api/<file>')` at the top of the test. See [`JsonEditorPage.test.tsx`](Frontend/src/test/JsonEditorPage.test.tsx) for the full pattern.
- **E2E**: Playwright tests in `Frontend/e2e/`. Base URL: `http://localhost:5173`.

## Agent Customization — Auto-Create Policy

**Always create or update customization files as part of the task. Never only suggest them.**

When any of the triggers below apply, create the file immediately alongside the code changes.

### When to create / update each type

| Trigger | File type | Location |
|---------|-----------|----------|
| A new library, framework, or package is integrated | `.instructions.md` | `.github/instructions/<topic>.instructions.md` |
| A repeatable multi-step workflow is introduced (new resource, new route, new component pattern) | `.prompt.md` | `.github/prompts/<task>.prompt.md` |
| A convention is discovered that differs from common practice | Add a rule to the relevant `.instructions.md` | — |
| An existing `applyTo` glob doesn't cover new file paths being created | Update `applyTo` in the relevant `.instructions.md` | — |
| A set of steps would benefit from agent isolation or tool restriction | `.agent.md` | `.github/agents/<name>.agent.md` |
| `AGENTS.md` is missing context an agent would need to be productive | Update `AGENTS.md` | root |

### Rules

- **Do it, don't mention it.** Create or edit the file in the same response as the code. Do not end a response with "you could create a prompt for this".
- **Link, don't duplicate.** Reference existing docs with Markdown links rather than copying their content into the new file.
- **Minimal and actionable.** Every line in a customization file must guide concrete behaviour. Remove vague or obvious content.
- **Keep `applyTo` tight.** Use specific globs (e.g. `src/hooks/**`, `src/routes/take/**`). Avoid `**` unless the instruction genuinely applies everywhere.
- **Prompts include a verification step.** Every `.prompt.md` that scaffolds code must end with a step that runs `pnpm build` (or `pnpm test`) and fixes any errors.
